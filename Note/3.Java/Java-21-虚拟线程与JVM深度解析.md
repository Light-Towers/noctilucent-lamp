# Java 高并发架构：从传统线程到虚拟线程 (Project Loom) 深度解析

## 1. 为什么需要虚拟线程？(The "Why")

### 1.1 传统“每请求一线程 (Thread-per-Request)”模型的困境
在 Java 21 之前，Java 的 `java.lang.Thread` 是**平台线程 (Platform Thread)**，它与操作系统的内核线程是 **1:1 映射**的。
- **内存昂贵**：每个平台线程默认分配 1MB 的栈空间，1000 个线程就占用 1GB 内存。
- **切换昂贵**：线程切换涉及内核态与用户态的上下文切换，CPU 周期消耗大。
- **吞吐量瓶颈**：面对海量 I/O 请求（如微服务调用），线程大都在等待 I/O，但操作系统无法支持开启几十万个物理线程。

### 1.2 异步/响应式编程的权宜之计
为了解决上述问题，业界推出了 Netty, Vert.x, WebFlux 等。它们通过非阻塞 I/O 提高了吞吐量，但代价极其惨重：
- **代码破碎**：业务逻辑被拆成无数个回调 (Callback)。
- **调试灾难**：堆栈跟踪 (Stack Trace) 丢失，无法一眼看出哪里出错。

## 2. 虚拟线程的核心原理 (How it Works)

虚拟线程（Virtual Threads）引入了 **M:N 调度模型**：
- **M (Virtual Threads)**：可以是几百万个，极其轻量（初始仅占几百字节到几 KB）。
- **N (Carrier Threads)**：底层的平台线程（通常等于 CPU 核心数），负责实际执行计算。

### 核心机制：挂起与恢复 (Mounting & Unmounting)
当虚拟线程执行到阻塞操作（如 `Thread.sleep` 或 Socket 读写）时：
1. JVM 会自动捕获这个阻塞调用。
2. 虚拟线程的堆栈信息会被从平台线程“搬”到堆内存中（**Unmount**）。
3. 平台线程立即去执行其他的虚拟线程。
4. 当 I/O 就绪，JVM 再把堆里的堆栈信息搬回平台线程继续执行（**Mount**）。
**这一切对开发者是透明的！你依然可以写简单的阻塞代码，却拥有异步的性能。**

## 3. JVM 视角下的变化

### 3.1 调度器 (Scheduler)
虚拟线程由 JVM 内部的 `ForkJoinPool` 进行调度，而不是操作系统的调度器。

### 3.2 锁固定 (Pinning) 问题 —— 2026 面试必考
在虚拟线程中，如果代码块使用了 `synchronized` 关键字，且内部发生了 I/O 阻塞：
- **现象**：虚拟线程无法被搬走，它会“死死粘”在底层的平台线程上。
- **后果**：如果所有平台线程都被固定住，整个程序就会阻塞，性能崩塌。
- **对策**：使用 `java.util.concurrent.locks.ReentrantLock` 替代 `synchronized`。

## 4. 实战准则 (Best Practices)

1.  **不要池化 (Don't Pool)**：以前我们需要线程池是因为线程贵；现在虚拟线程比对象还便宜，用完即丢，千万不要用 `ExecutorService` 去缓存它们。
2.  **保持同步编码风格**：忘掉 `CompletableFuture` 和响应式框架，回归最简单的顺序逻辑。
3.  **限制并发资源**：如果下游数据库只能承载 100 个连接，不要因为能开 10 万个虚拟线程就猛冲数据库，应改用 `Semaphore` 限制并发。
