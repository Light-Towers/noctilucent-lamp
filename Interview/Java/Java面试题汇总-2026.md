- **栈帧结构？**
  - 局部变量表
  - 操作数栈
  - 动态链接
  - 方法返回地址
- **方法区存储什么？**

- **直接内存？**

### 2. 对象的创建与内存布局

- **对象的访问定位？**

### 3. 垃圾回收 ⭐⭐⭐
- **如何判断对象可以被回收？**
  - 引用计数法（循环引用问题）
  - 可达性分析算法（GC Roots）
- **GC Roots有哪些？**
  - 虚拟机栈中引用的对象
  - 方法区中静态属性引用的对象
  - 方法区中常量引用的对象
  - 本地方法栈中JNI引用的对象
- **Java的引用类型？**
  - 强引用（Strong Reference）
  - 软引用（Soft Reference）：内存不足时回收
  - 弱引用（Weak Reference）：GC时回收
  - 虚引用（Phantom Reference）：跟踪回收
- **垃圾回收算法？**
  - 标记-清除（Mark-Sweep）：效率低，内存碎片
  - 标记-复制（Mark-Copy）：效率高，内存利用率低
  - 标记-整理（Mark-Compact）：效率低，无碎片
- **分代收集理论？**
  - 新生代：复制算法
  - 老年代：标记-清除或标记-整理
- **垃圾收集器？**
  - Serial：单线程，新生代
  - ParNew：多线程，新生代
  - Parallel Scavenge：多线程，新生代，关注吞吐量
  - Serial Old：单线程，老年代
  - Parallel Old：多线程，老年代
  - CMS（Concurrent Mark Sweep）：老年代，关注低延迟
    - 初始标记（STW）
    - 并发标记
    - 重新标记（STW）
    - 并发清除
    - 问题：CPU敏感、浮动垃圾、空间碎片
  - G1（Garbage First）：面向服务端
    - 将堆划分为多个Region
    - Young GC、Mixed GC、Full GC
    - 可预测停顿时间
  - ZGC（JDK11）：低延迟，停顿时间不超过10ms
  - Shenandoah（JDK12）：低延迟

### 4. 类加载机制 ⭐⭐
- **类加载的过程？**
  1. 加载（Loading）
  2. 验证（Verification）
  3. 准备（Preparation）
  4. 解析（Resolution）
  5. 初始化（Initialization）
  6. 使用（Using）
  7. 卸载（Unloading）
- **类加载器有哪些？**
  - 启动类加载器（Bootstrap ClassLoader）
  - 扩展类加载器（Extension ClassLoader）
  - 应用程序类加载器（Application ClassLoader）
  - 自定义类加载器
- **双亲委派模型？**
  - 原理：先委托父加载器加载，父加载器无法加载时自己加载
  - 好处：避免类重复加载、保护核心类安全
  - 如何打破双亲委派？
    - 重写loadClass()方法
    - 线程上下文类加载器
    - OSGi模块化

### 5. JVM调优 ⭐⭐
- **常用的JVM参数？**
  - -Xms：初始堆大小
  - -Xmx：最大堆大小
  - -Xmn：新生代大小
  - -XX:NewRatio：新生代与老年代比例
  - -XX:SurvivorRatio：Eden与Survivor比例
  - -XX:MetaspaceSize：元空间大小
  - -XX:MaxMetaspaceSize：最大元空间大小
  - -XX:+UseSerialGC：使用Serial收集器
  - -XX:+UseParallelGC：使用Parallel收集器
  - -XX:+UseConcMarkSweepGC：使用CMS收集器
  - -XX:+UseG1GC：使用G1收集器
  - -XX:MaxGCPauseMillis：G1最大停顿时间
- **JVM调优工具？**
  - jps：查看Java进程
  - jstat：查看GC统计信息
  - jmap：生成堆转储快照
  - jstack：生成线程转储快照
  - jinfo：查看JVM配置信息
  - jconsole：图形化监控工具
  - jvisualvm：图形化监控工具
  - MAT：内存分析工具
  - Arthas：阿里开源诊断工具

### 6. 内存溢出与内存泄漏
- **内存溢出（OOM）？**
  - Java heap space：堆溢出
  - Metaspace：元空间溢出
  - GC overhead limit exceeded：GC时间过长
  - Direct buffer memory：直接内存溢出
- **内存泄漏？**
  - 静态集合类
  - 未关闭的连接
  - ThreadLocal未remove
  - 监听器未注销
- **如何排查OOM？**
  - jmap生成堆转储
  - MAT分析堆转储
  - 找到占用内存最大的对象

### 3. Spring事务
- **Spring事务的实现原理？**（AOP）

  - SUPPORTS：支持当前事务，不存在则以非事务执行
  - MANDATORY：支持当前事务，不存在则抛异常
  - REQUIRES_NEW：新建事务，挂起当前事务

  - REPEATABLE_READ

  - 异常被catch捕获
  - 抛出检查异常（默认只回滚运行时异常）
  - 数据库不支持事务

## 八、数据库 ⭐⭐⭐

### 1. MySQL索引
- **索引的数据结构？**
  - B+树：MySQL默认
  - Hash：Memory引擎
- **为什么使用B+树？**
  - 层级少，查询快
  - 叶子节点链表，范围查询快
  - 节点只存键值，单节点存更多元素
- **索引类型？**
  - 聚簇索引：主键索引，叶子节点存整行数据
  - 非聚簇索引：二级索引，叶子节点存主键值
  - 联合索引：多个列的索引
  - 唯一索引：列值唯一
  - 全文索引：文本搜索
- **索引优化？**
  - 最左前缀原则
  - 覆盖索引
  - 索引下推（ICP）
  - 避免索引失效
    - 避免在索引列上计算
    - 避免隐式转换
    - 避免使用!=、<>、NOT IN
    - 避免使用OR
    - LIKE以%开头

### 2. MySQL事务
- **事务的ACID特性？**
  - Atomicity（原子性）
  - Consistency（一致性）
  - Isolation（隔离性）
  - Durability（持久性）
- **事务隔离级别？**
  - READ UNCOMMITTED：读未提交
  - READ COMMITTED：读已提交（Oracle默认）
  - REPEATABLE READ：可重复读（MySQL默认）
  - SERIALIZABLE：串行化
- **并发事务的问题？**
  - 脏读
  - 不可重复读
  - 幻读
- **MVCC（多版本并发控制）？**
  - Read View
  - Undo Log
  - 隐藏字段：trx_id、roll_pointer

### 3. MySQL锁
- **锁的类型？**
  - 表锁：开销小，并发低
  - 行锁：开销大，并发高
  - 页锁：介于表锁和行锁之间
- **行锁算法？**
  - Record Lock：记录锁
  - Gap Lock：间隙锁
  - Next-Key Lock：记录锁+间隙锁
- **乐观锁和悲观锁？**

### 4. MySQL日志
- **Redo Log（重做日志）？**
  - 保证持久性
  - 顺序写入，性能高
- **Undo Log（回滚日志）？**
  - 保证原子性
  - MVCC的基础
- **Bin Log（归档日志）？**
  - 主从复制
  - 数据恢复

### 5. SQL优化
- **如何分析SQL性能？**
  - EXPLAIN
  - SHOW PROFILE
  - 慢查询日志
- **EXPLAIN关键字段？**
  - id：执行顺序
  - type：访问类型（ALL、index、range、ref、eq_ref、const、system）
  - key：使用的索引
  - rows：扫描行数
  - Extra：额外信息
- **优化建议？**
  - 避免SELECT *
  - 使用LIMIT
  - 合理使用索引
  - 避免大事务
  - 分页优化

### 6. Redis ⭐⭐
- **Redis的数据类型？**
  - String
  - List
  - Set
  - Hash
  - ZSet（有序集合）
  - Bitmap
  - HyperLogLog
  - Geo
  - Stream（Redis 5.0）
- **Redis为什么快？**
  - 基于内存
  - 单线程，避免上下文切换
  - IO多路复用
  - 高效的数据结构
- **Redis的持久化？**
  - RDB：快照，文件小，恢复快，可能丢数据
  - AOF：追加日志，数据安全，文件大
  - 混合持久化
- **Redis的过期策略？**
  - 定期删除
  - 惰性删除
- **Redis的内存淘汰策略？**
  - noeviction：不淘汰
  - allkeys-lru：从所有键中淘汰最近最少使用
  - volatile-lru：从设置了过期时间的键中淘汰
  - allkeys-random：随机淘汰
  - volatile-random：从设置了过期时间的键中随机淘汰
  - volatile-ttl：淘汰TTL最小的
- **Redis的集群？**
  - 主从复制
  - 哨兵（Sentinel）
  - Cluster（分片）
- **缓存问题？**
  - 缓存穿透：查询不存在的数据
    - 解决：布隆过滤器、缓存空值
  - 缓存击穿：热点key过期
    - 解决：互斥锁、永不过期
  - 缓存雪崩：大量key同时过期
    - 解决：随机过期时间、高可用
- **Redis分布式锁？**（见分布式锁章节）
